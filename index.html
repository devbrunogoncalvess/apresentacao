PRs criados manualmente

Pull Requests feitos diretamente na master sem um processo padronizado aumentam o risco de erros humanos.

📉

Sem automação de CD

Apenas o CI está automatizado. O processo de entrega depende de ações manuais e não tem auditoria clara.

Proposta
A Solução Proposta
🚀

Pipeline Inteligente com Aprovação Humana

Automatizar tudo que pode ser automatizado — criação de PRs, atualização de Work Items e rastreabilidade — mantendo o controle humano apenas no ponto crítico: a aprovação antes do merge na master.

01

Service Hook

Gatilho automático quando um Work Item muda para "Aguardando Implementação".

02

Auto PR via REST API

Pipeline cria o Pull Request da branch correta para a master automaticamente.

03

Update Automático

Após o merge, o Work Item é atualizado para "Publicado" sem intervenção humana.

Arquitetura
As 3 Peças da Solução
🔔

Service Hook

Gatilho configurado no Azure DevOps que escuta mudanças de estado dos Work Items.

  • Evento: Work Item Updated
  • Filtro: State = Aguardando Implementação
  • Ação: Dispara o pipeline de PR
  • Sem código adicional necessário
📄

Pipeline Auto-PR (YAML)

Pipeline que recebe o ID do Work Item e cria o PR automaticamente via REST API.

  • Recebe workItemId e branchName
  • Autentica via System.AccessToken
  • Cria PR com título e descrição padrão
  • Vincula o PR ao Work Item
✅

Pipeline Update WI (YAML)

Pipeline acionado pelo merge na master que atualiza o status do Work Item.

  • Trigger: push na master
  • Busca WI vinculado ao commit
  • Atualiza estado para Publicado
  • Log auditável de cada publicação

Código
Pipeline: Criação Automática de PR
azure-pipelines-auto-pr.yml
# Pipeline disparado via Service Hook quando WI → Aguardando Implementação
trigger: none

parameters:
  - name: workItemId
    type: string
  - name: branchName
    type: string

variables:
  ORGANIZATION: 'sua-organizacao'
  PROJECT:      'seu-projeto'
  REPO:         'seu-repositorio'

stages:
  - stage: CreatePR
    jobs:
      - job: AutoPR
        pool:
          vmImage: 'ubuntu-latest'
        steps:
          - task: PowerShell@2
            inputs:
              targetType: 'inline'
              script: |
                # Configura headers de autenticação
                $headers = @{
                  Authorization  = "Bearer $(System.AccessToken)"
                  "Content-Type" = "application/json"
                }
                # Monta o body do PR com vínculo ao Work Item
                $body = @{
                  title         = "Auto PR - WI #$(workItemId)"
                  sourceRefName = "refs/heads/$(branchName)"
                  targetRefName = "refs/heads/master"
                  workItemRefs  = @(@{ id = "$(workItemId)" })
                } | ConvertTo-Json
                # Cria o PR via REST API
                $url = "https://dev.azure.com/$(ORGANIZATION)/$(PROJECT)/_apis/git/repositories/$(REPO)/pullrequests?api-version=7.1"
                Invoke-RestMethod -Uri $url -Method POST -Headers $headers -Body $body

Pipeline: Atualização do Work Item
azure-pipelines-update-workitem.yml
# Pipeline disparado após merge na master → atualiza WI para "Publicado"
trigger:
  branches:
    include:
      - master

variables:
  ORGANIZATION: 'sua-organizacao'
  PROJECT:      'seu-projeto'

stages:
  - stage: UpdateWorkItem
    jobs:
      - job: UpdateWI
        pool:
          vmImage: 'ubuntu-latest'
        steps:
          - task: PowerShell@2
            inputs:
              targetType: 'inline'
              script: |
                $headers = @{
                  Authorization  = "Bearer $(System.AccessToken)"
                  "Content-Type" = "application/json"
                }
                # Busca Work Items vinculados ao commit do merge
                $commitId  = "$(Build.SourceVersion)"
                $repoId    = "$(Build.Repository.ID)"
                $urlCommit = "https://dev.azure.com/$(ORGANIZATION)/$(PROJECT)/_apis/git/repositories/$repoId/commits/$commitId/workItems?api-version=7.1"
                $workItems = Invoke-RestMethod -Uri $urlCommit -Method GET -Headers $headers

                foreach ($wi in $workItems.value) {
                  # Atualiza estado do Work Item para "Publicado"
                  $bodyUpdate = @(
                    @{ op = "add"; path = "/fields/System.State"; value = "Publicado" }
                  ) | ConvertTo-Json
                  $urlUpdate = "https://dev.azure.com/$(ORGANIZATION)/$(PROJECT)/_apis/wit/workitems/$($wi.id)?api-version=7.1"
                  Invoke-RestMethod -Uri $urlUpdate -Method PATCH -Headers $headers -Body $bodyUpdate -ContentType "application/json-patch+json"
                  Write-Host "✅ Work Item #$($wi.id) → Publicado"
                }

Migrations Automatizadas

Controle de versão do banco de dados integrado ao pipeline

🗃️ Flyway 🐘 PostgreSQL 📋 .NET / C# 🔁 Multi-ambiente
Diagnóstico
O Problema com as Migrations
🔀
Ordem Incorreta

Scripts fora de ordem

Scripts SQL versionados manualmente não garantem a ordem de execução, causando falhas em cascade ou dados inconsistentes.

❓
Sem Histórico

Não sabe o que foi aplicado

Sem registro centralizado, é impossível saber quais scripts já foram executados em cada ambiente (dev, homolog, prod).

🖐️
Processo Manual

Execução manual por ambiente

Alguém precisa acessar cada banco e executar os scripts manualmente, gerando risco de pular etapas ou aplicar duplicado.

🔗
Desconexão

Sem integração com o pipeline

As migrations de banco acontecem de forma independente do deploy da aplicação, criando janelas de incompatibilidade.

Solução
Por que Flyway?
🌱

Controle de versão para o banco de dados

Flyway mantém um histórico completo de quais scripts foram aplicados em cada banco, garante a ordem correta de execução e integra nativamente com Azure DevOps Pipelines e .NET/C#.

V1__

Ordem garantida

Prefixo de versão define a sequência de execução automaticamente

📋

Histórico no banco

Tabela flyway_schema_history registra tudo com checksum e data

🔒

Idempotente

Nunca executa o mesmo script duas vezes — verifica checksum antes

🔁

Multi-ambiente

Configurações separadas para dev, homolog e prod no mesmo projeto

Organização
Estrutura de Pastas e Nomenclatura
📁 /db 📁 /migrations 📄 V1__cria_tabela_usuarios.sql 📄 V2__adiciona_coluna_email.sql 📄 V3__cria_tabela_pedidos.sql 📄 V4__index_pedidos.sql 📄 V5__nova_feature.sql ← nova 📁 /flyway ⚙️ flyway-dev.conf ⚙️ flyway-homolog.conf ⚙️ flyway-prod.conf

📐 Regra de nomenclatura

V{número}__ → versão obrigatória
Dois underscores → separador obrigatório
Descrição → use snake_case legível
.sql → extensão padrão

Código
Pipeline: Flyway Multi-Ambiente
azure-pipelines-migrations.yml
# Roda migrations automaticamente por branch
trigger:
  branches:
    include: [ develop, homolog, master ]

variables:
  FLYWAY_VERSION: '10.10.0'

stages:
  # ── DEV ─────────────────────────────────────────
  - stage: Migration_Dev
    condition: eq(variables['Build.SourceBranch'], 'refs/heads/develop')
    jobs:
      - job: RunDev
        pool: { vmImage: 'ubuntu-latest' }
        steps:
          - script: |
              curl -L https://repo1.maven.org/.../flyway-commandline-$(FLYWAY_VERSION)-linux-x64.tar.gz | tar xz
              ./flyway-$(FLYWAY_VERSION)/flyway -configFiles=./db/flyway/flyway-dev.conf migrate
            env: { DB_PASSWORD: $(DB_PASSWORD_DEV) }

  # ── HOMOLOG ─────────────────────────────────────
  - stage: Migration_Homolog
    condition: eq(variables['Build.SourceBranch'], 'refs/heads/homolog')
    jobs:
      - job: RunHomolog
        pool: { vmImage: 'ubuntu-latest' }
        steps:
          - script: |
              curl -L .../flyway-commandline-$(FLYWAY_VERSION)-linux-x64.tar.gz | tar xz
              ./flyway-$(FLYWAY_VERSION)/flyway -configFiles=./db/flyway/flyway-homolog.conf migrate
            env: { DB_PASSWORD: $(DB_PASSWORD_HOMOLOG) }

  # ── PROD (com aprovação humana) ──────────────────
  - stage: Migration_Prod
    condition: eq(variables['Build.SourceBranch'], 'refs/heads/master')
    jobs:
      - deployment: RunProd
        environment: 'producao'   # ← gate de aprovação
        strategy:
          runOnce:
            deploy:
              steps:
                - script: |
                    ./flyway-$(FLYWAY_VERSION)/flyway -configFiles=./db/flyway/flyway-prod.conf migrate
                  env: { DB_PASSWORD: $(DB_PASSWORD_PROD) }

Arquitetura
As Peças da Solução Flyway
🗂️

Scripts Versionados

Nomenclatura padronizada garante ordem e rastreabilidade.

  • Prefixo V{n}__ obrigatório
  • Nomes descritivos em snake_case
  • Versionados junto ao código no Git
  • Revisados via PR como qualquer código
⚙️

Configs por Ambiente

Cada ambiente tem sua configuração isolada e segura.

  • flyway-dev.conf
  • flyway-homolog.conf
  • flyway-prod.conf
  • Senhas via variáveis do pipeline
📊

Histórico no Banco

Flyway mantém tabela de auditoria automática no PostgreSQL.

  • flyway_schema_history gerada automaticamente
  • Registra versão, descrição e checksum
  • Data e usuário de cada execução
  • Detecta scripts alterados indevidamente
